# 優先度【高】

### 1. ステージCの「調べる」メニュー(8項目)がシーン領域からはみ出し、先頭項目が欠ける

- **対象**: HTML版 ／ C:\Users\user\Downloads\ame_no_owarini.html 行21-22(#stage aspect-ratio:760/300; overflow:hidden), 行39-41(#choice bottom:8px gap:5px), 行800-810(書斎ステージCで7項目), 行981-984(doExamineが「やめる」を追加し計8項目)
- **根拠**: 実測(ブラウザで本ファイルを描画して計測): 選択肢ボタン1個=35px高(font 14.5px+padding 5px×2+border 1px×2, Yu Minchoのline-height normal)。8項目メニュー=8×35+7×5(gap)=315px。一方#stageの高さは最大幅780pxでも307.9pxで、bottom:8px起点のため上端が16.1pxはみ出し、overflow:hiddenでクリップされる。ウィンドウ幅700pxでは stage=265.3px となり58.8pxクリップ=先頭「机の上（封筒）」が完全消失+2番目「机の引き出し」も上24px欠け。幅683px(1366px画面の半分スナップ)では2番目の可視部が約11pxまで減る。
- **影響**: ステージCで全員が必ず開くメニューで、最良条件でも先頭ボタンの上半分が欠けて表示が破綻する。小さめウィンドウ(ハーフスナップ等)では進行必須の証拠「机の引き出し」(創作ノート、行1032のトライアル開始条件)の項目自体が見えなくなり、実質ソフトロック級になる。
- **修正案**: #choiceに max-height:calc(100% - 16px); overflow-y:auto を追加するのが最小修正。より良いのは項目数が6を超えたら2列グリッドに切り替えるか、ボタンのpadding/フォントを縮小して8項目が300px内に収まるよう設計すること。

### 2. Ren'Py版: ロールバック有効のままライフ制を実装しており、ホイール1段で誤答ペナルティを取り消せる

- **対象**: Ren'Py版 ／ C:\Users\user\Downloads\ame-no-owarini\renpy\AmeNoOwariNi\game\options.rpy 20-23行, trial.rpy 119-128行
- **根拠**: options.rpy:22のコメント「読み返しはマウスホイール（ロールバック）でできる」の通りロールバックは有効化されたまま(game配下に block_rollback / fix_rollback の呼び出しは0件、grepで確認)。wrong_present(trial.rpy:120 `$ lives -= 1`)のlivesは通常のstore変数なので、誤答直後にホイールを上に1段回すだけで減点前の状態に戻る。逆に証言ナビ中に誤ってホイールを回すと、call_screenのインタラクションを跨いで意図せず巻き戻る。
- **影響**: ライフ5・ゲームオーバー再挑戦という緊張感の設計が、知っている人には完全に無効化される。知らない人には誤ホイールで尋問が勝手に巻き戻る事故になる。読み返し手段をロールバックに一本化した設計判断と、ライフ制が正面衝突している。
- **修正案**: wrong_present内とラウンド正解直後に `$ renpy.block_rollback()` を入れて減点を確定させる。読み返しはバックログ画面(screen + _history_list)を別途用意すると両立できる。

### 3. Ren'Py版は調査コマンドが画面中央に出てシーンを覆い、その間メッセージ窓が消えて下部215pxが黒帯になる

- **対象**: Ren'Py版 ／ C:\Users\user\Downloads\ame-no-owarini\renpy\AmeNoOwariNi\game\screens.rpy:58-65
- **根拠**: screen choice(58-65行)は `xalign 0.5 / yalign 0.40 / xminimum 460` で画面中央(720pxの40%≒288px付近)に表示。背景は上部505pxのみ(defs.rpy 11行コメント、say_window は ypos 505 ysize 215 = screens.rpy 22-28行)なので、menu中はsay窓が非表示になり下部215pxが無地の黒になる。HTML版は逆にコマンドバー #bar(67-68行)がメッセージ窓の「下」に常設され(1084-1088行 waitCommand)、直前のテキストを表示したままシーンを一切隠さない。神宮寺/ファミ探系の慣習もコマンド欄は下部/隅固定。
- **影響**: 調査ループは数十回メニューを開くため、そのたびに背景中央が覆われ「いま読んだ文」も消える。現場の絵を見ながらどこを調べるか考える、というコマンドADVの基本動線が崩れる。
- **修正案**: style choice の vbox を `yalign 0.93` 程度まで下げ(HTML版の bottom:8px 相当)、options.rpy に `define config.window = "auto"` を追加するか各 menu の直前に `window show` を置いて直前テキストを保持する。

# 優先度【中】

### 4. メッセージウィンドウをクリックしても読み進められない(送りクリック領域がシーン部のみ)

- **対象**: HTML版 ／ C:\Users\user\Downloads\ame_no_owarini.html 行30(#clicker{position:absolute;inset:0}), 行105(#clickerは#stage内), 行121-125(#msgは#stage外の兄弟要素), 行64-65(▼カーソルは#msg内 right:14px bottom:6px)
- **根拠**: 実測: クリック送り可能な#clickerは778×306px(シーン領域のみ)。テキストが表示される#msgは780×122pxでその直下にあり、クリックリスナは無い。さらに送り待ちの▼カーソルはその#msgの右下で点滅する(行64-65)。つまり視線とカーソル誘導の先がクリック無効地帯。タイトル画面の説明は「クリック／Enterで読み進める」(行112)でクリック位置を限定していない。
- **影響**: ADVの慣習(テキストボックスをクリックして送る)に従ったプレイヤーの最初のクリックが無反応になり、「固まった?」と誤解させる。▼が無反応領域で点滅し続けるため、約10分間ずっと小さな摩擦として残る。
- **修正案**: クリックリスナを#gameに付け替える(またはdocumentでclicker表示中のみ処理)か、#clickerを#game直下に移して inset:0 で#msg・#barまで覆う。選択肢やモーダル表示中はz-indexで既に保護されているため副作用は小さい。

### 5. キーボード操作がEnter/Spaceの読み進めのみで、尋問・選択肢・手帳で全滅する

- **対象**: HTML版 ／ C:\Users\user\Downloads\ame_no_owarini.html 行644-648(keydownはEnter/Spaceのみ、しかもclicker表示中限定), 行1098-1103(showStatement末尾でclickerOn(false)), 行1092-1096(尋問の4アクションはバーのボタンのみ), 行679-690(選択肢もマウスのみ), 行752(手帳はボタンclickのみ)
- **根拠**: keydownハンドラは1箇所だけで `if(e.key==='Enter'||e.key===' '){ if(clicker.style.display==='block'){...} }` (行644-648)。Escはどこにもバインドされておらず手帳モーダル(#modal)はEnterでもEscでも閉じない。尋問では証言表示完了後にclickerOn(false)(行1103)が呼ばれるためEnter/Spaceすら無効になり、「◀前の証言/次の証言▶/ゆさぶる/つきつける」は矢印キー非対応のマウス専用ボタン(行1095)。選択肢メニューにも数字キー/矢印キーのフォーカス移動はない(Tab+Enterのブラウザ標準のみ)。
- **影響**: タイトルの案内(行112「クリック／Enterで読み進める」)に従ってキーボードで読んでいたプレイヤーが、選択肢・尋問・手帳の度にマウスへ持ち替えを強制される。特に尋問中はEnterが突然死ぬため「キーが効かなくなった」と感じる。逆裁オマージュなのに←→で証言移動できないのは期待とのギャップが大きい。
- **修正案**: keydownに追加: Escで nbFinish(null)(select中は「やめる」相当)、尋問バー表示中は ←/→ を prev/next、選択肢表示中は数字キー1〜9で choose を確定。実装はグローバルkeydown1箇所に現在の待ち受け状態(advResolve/choice/bar/modal)で分岐させれば30行程度。

### 6. game_menu空化により右クリック・Escが完全無反応で、音量・フルスクリーン・終了に到達する手段がゲーム内にない

- **対象**: Ren'Py版 ／ C:\Users\user\Downloads\ame-no-owarini\renpy\AmeNoOwariNi\game\options.rpy:20-23
- **根拠**: L23 `config.keymap['game_menu'] = []` は、Ren'Py既定でgame_menuに束ねられている K_ESCAPE / mouseup_3(右クリック) / pad_back_press 等を全て無効化する。代替バインドは全.rpyに存在せず(grepでShowMenu/Preference系アクションは0件、key文はscreens.rpy L186-187の2つのみ)、終了手段はウィンドウ×のquit_action(options.rpy L18)だけ。音量はL14-15で初期値0.8固定のまま変更UIがない。
- **影響**: Ren'Pyの習慣で右クリックやEscを押したプレイヤーには一切反応がなく、フリーズや未実装と誤認される。雨音ループ+BGMの音量をゲーム内で下げられず、フルスクリーン切替(Alt+Enterは生きているが)の案内もない。
- **修正案**: 「再開/音量/最初からやり直す/終了」だけの小さなポーズ画面を自作し、keymapは既定のまま `define _game_menu_screen = "pausescr"` で割り当てる。最低でも右クリック/Escに終了確認(quit_actionと同じConfirm)を紐づけ、無反応をなくす。

### 7. ホイールロールバックで尋問のライフが回復し、ペナルティ制を総当たりで無効化できる

- **対象**: Ren'Py版 ／ C:\Users\user\Downloads\ame-no-owarini\renpy\AmeNoOwariNi\game\trial.rpy:119-128
- **根拠**: wrong_presentのL120 `$ lives -= 1` の直後、L124-126は通常のsay表示でありホイール上のロールバックが有効(options.rpy L21-22のコメント通り、ロールバックは意図的に許可されている)。livesはdefs.rpy L36で `default lives = 5` と宣言されたストア変数なのでロールバック対象で、巻き戻すと減算前の値に戻る。renpy.block_rollback()/fix_rollbackの呼び出しはgameディレクトリ全体で0件(grep確認)。gameover_seq(L130-141)も同様に巻き戻して回避できる。
- **影響**: 「見当違いの証拠でライフ(◆)が減る」(trial.rpy L58で明示)という尋問パートの緊張感の核が、ホイール1回で骨抜きになる。気づいたプレイヤーは全証拠の総当たりで解けてしまい、ゲームオーバー演出もほぼ到達不能の死にコンテンツになる。

## 検証で棄却された指摘（誤検知）

- **音トグルのラベル「音：オン」が状態表示か操作かが曖昧で、ONに戻した時の確認音も無い**
  - 棄却理由: 引用自体は正確(行99に「音：オン」、行759-763にsfx無しのハンドラ、行157にsfxSel実在、76×28pxも.hbtn定義(行20)から妥当)。しかし主張の核心である「オンに戻しても音で判別できない→往復誤操作」はコードと矛盾する。トグルはAC全体のsuspend/resume方式(行762)であり、雨音はループ再生の連続ノイズ源(行180-191、src.loop=true、gain.045)としてタイトル(行778 setRain(true,2))からエピローグ(行1280)まで常時鳴っているため、オンに戻した瞬間に雨音が即座に復帰し、それ自体が確認音として機能する。BGMシーケンサも110ms周期(行235-244)で即復帰し、唯一のsetBGM(null)区間(行1141)でも雨音は継続、雨停止後(行1280以降)はsadパターン(行227-231、全16ステップに音符)が連続再生。ゲーム中に「オンにしたのに無音」となる状態は存在せず(AC未生成のタイトル初期のみだがその時点では音自体未開始)、指摘の影響シナリオは発生しない。残るのは「音：オン」のコロン形式ラベルの抽象的な曖昧性のみだが、これは状態表示の慣習的書式であり、押せば1秒未満で聴覚フィードバックにより自己解決する。別の仕組みで既に解決済みと判断しreal=false。
- **modal Trueの証言画面・手帳上ではホイール読み返しが効かず、options.rpyの「読み返しはホイールで」という設計前提が尋問中に崩れる**
  - 棄却理由: 引用行は実在する(screens.rpy L92/L170のmodal True、options.rpy L21-23のコメント、trial.rpy L76/L79/L83の% len(_rd["sts"])、L243のR1参照台詞)が、指摘の中核機構「modal画面がconfig.underlayのロールバックを遮断する」がRen'Pyの実装と食い違う。ユーザーのSDK(C:\Users\user\Downloads\renpy-sdk)で確認: (1)doc\screens.htmlのmodal解説に「except for the default keymap」と明記。(2)renpy\display\screen.py L812-813でmodal画面はIgnoreLayersをraiseするが、layout.py L588-592のdocstringが「ignored by layers, but reach the underlay」と明言し、L1285-1290でlayersルートが捕捉してNoneを返すため、core.py L2505-2508でroot_widget直下(レイヤー外)に追加されたunderlayへイベントが到達する。(3)common\00keymap.rpy L39でrollback=mousedown_4(ホイール上)、L437/L466でそのKeymapがconfig.underlay。よって証言・手帳のmodal画面表示中もホイールロールバックは機能し、call_screenはチェックポイント対象、ゲーム側にrollback無効化記述もなし(全.rpyをgrepで確認、0件)。R2からR1証言まではデフォルトhard_rollback_limit=100以内で巻き戻し可能。「前へ/次へがラウンド内循環のみ」「履歴画面なし」は事実だが、「確認し直す手段が一切ない」という影響主張はホイール読み返しが効く以上成立しない。
- **Ren'Py版: セルフボイシングで証言文・手帳の説明文・入手カードが読み上げられない**
  - 棄却理由: 引用行は実在(screens.rpy 169-176のtext stxt、137のev_desc、162-165のshort、209-211のタイトル文)。しかし指摘の前提「セルフボイシングはsayとフォーカス要素しか読まない」がRen'Pyの仕様と食い違う。公式ドキュメント(self_voicing.html)は「If no displayable is focused, Ren'Py will read on-screen text that cannot be focused」と明記し、renpy/display/tts.pyのdisplayable()はd=Noneでroot._tts_all()を読み上げ、Displayable._ttsは「フォーカス不能な子のテキストを返す」仕様を確認。本作はクイックメニュー未定義のためsayのセリフ自体もこの「無フォーカス時のroot読み上げ」で読まれており、testimonyのtext stxt・notebookのdesc・タイトル文も同一経路で読み上げ対象になる。さらに尋問は行186-187のkey K_LEFT/K_RIGHTで前後移動でき、キーボード操作では各証言が新インタラクション開始時に自動読み上げされる。残る真実の核は(1)マウスがボタン上に乗ったままだとフォーカス要素のみ読まれ新証言が自動では読まれない(ポインタを空白へ動かせば全文読み上げ)、(2)itemgetの全画面ボタン(行152-156)が常時フォーカスを奪い入手カード文が自動で読まれない――の2点だが、いずれも「読み上げ対象にならない」「実質プレイ不能」という指摘内容とは別物で、影響は軽微。指摘の核心的メカニズムが誤りのためreal=false。
